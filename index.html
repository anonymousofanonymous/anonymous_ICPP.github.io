Response to Reviewer #1

Thank you very much for your comments. We appreciate your time and effort in helping us improve the manuscript! Please see below our response.

Comment #1: Do you have a way to measure the absolute efficiency? Speedups are great but is this an efficient implementation or not?

Response #1: Thank you for the comment. The absolute efficiency can be observed at the two break-even points, 256 and 1024 stimulus, in Table 2 for Spinal and NVDLA. Similar number is observed in Figure 12 for riscv-mini. When the number of stimulus is small (i.e., <256), RTLflow does not benefit from much data parallelism and thus CPU-based Verilator is better. However, industrial simulators can easily call multi-millions of stimulus where RTLflow (GPU) wins out. We will add the two break-even points in section 4.3 to highlight the absolute efficiency.

 

Comment #2: What are the architectural bottlenecks?

Response #2: Thank you for the comment. RTLflow is primarily bottlenecked by the CPU-based call to set simulation input, i.e., dut.set_input in Listing 2. This is a constraint induced by the industrial convention where testbench drivers are typically written on CPU to read inputs from an external file and to handle control flow. However, as shown in Table 5, our pipeline can largely mitigate this challenge. We will revise Section 2.4.3 and Section 4.5 to highlight our explanations above.

Response to Reviewer #3

Thank you very much for your comments. We appreciate your time and effort in helping us improve the manuscript! Please see below our response.

Comment #1: One of the proposed optimizations is storing data in different types of arrays that would fit the data type size. However there are different coalescing patterns depending on the amount of data and cache lines have a fixed amount. I am not convinced that the strategy will yield the expected results. Similarly when using those different data types for operation some casting will be required.

Response #1: Thanks for so much for your comment. Yes, you are correct that different coalescing patterns depend on the number of stimulus. However, our transpliation does not impose any constraint on this and can be easily extended to handle the issue through parametertization. For example, our pipeline scheduling algorithm partitions all stimulus into groups so that all stimulus within a group is simultaneously simulated on a GPU. We can ensure memory access is mostly coalesced by setting the proper group size (e.g., 1024 stimulus per group).

 

Comment #2: In listing 3 a uint16 operand is stored into an uint8 data type which seems incorrect.

Response #2: Thanks for so much for your comment. The example code shown in Listing 3 is correct since a hardware design is allowed to copy a wider value to a narrower target. The purpose of Listing 3 is to show the difference between Verilator and RTLflows transpiled code. We use the same variable names as Figure 4, 6, and 7 to make variables consistent with Section 3.1. However, we understand your comment and will revise Listing 3 by using another example to remove confusion.

 

Comment #3: In the experimental results section the -O2 is used for the compilation of RTLFlow. Why not using -O3?

Response #3: Thank you for the comment. We did not observe much performance difference between -O2 and -O3, but -O2 makes the compilation time faster. We will add the explanation to Section 4 and the Appendix.
